# bet-loader

A personal iOS app that syncs your bet history from FanDuel, Caesars, and DraftKings into
a private Neon Postgres database — no backend server required.

Bets are harvested by loading each sportsbook's own web app inside an on-device WKWebView
and injecting JavaScript to intercept the SPA's authenticated API calls. Anti-bot measures
are sidestepped because the requests originate from a real browser on a real device using
the user's own logged-in session.

**Stack:** Expo Go → WKWebView → injected JS → sportsbook API → Neon Data API → Neon Postgres

## 1. Set up a fresh Neon project

### a. Create the project

Open [console.neon.tech](https://console.neon.tech) and create a new project. Note the
**production** branch — all steps below target that branch.

### b. Create the schema

Open the **SQL Editor** in the Neon Console (or connect with `psql` as the owner role) and
run `db/schema.sql`:

```sql
CREATE TABLE bets (
  id               bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id          text        NOT NULL DEFAULT (auth.user_id()),
  site_key         text        NOT NULL,
  external_bet_id  text        NOT NULL,
  placed_at        timestamptz,
  status           text,
  stake            numeric(12,2),
  potential_payout numeric(12,2),
  selections       jsonb,
  raw              jsonb,
  scraped_at       timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, site_key, external_bet_id)
);
```

> `auth.user_id()` is a Neon-provided SQL function that returns the `sub` claim from the
> current request's JWT. It requires Neon Auth to be enabled (next step).

### c. Enable Neon Auth

Go to **Auth** in the Neon Console sidebar and click **Enable**. This provisions a managed
Better Auth instance at `<your-auth-url>/auth`. Neon Auth issues JWTs whose `sub` claim
uniquely identifies each user — this is what drives `auth.user_id()` in Postgres.

### d. Enable the Data API

Go to **Data API** in the sidebar and click **Enable Data API**.

- Check **Use Neon Auth** — this wires your Auth endpoint as the JWT provider.
- Check **Grant public schema access** — this applies the base `GRANT` statements to the
  `authenticated` role automatically.

Once enabled, copy:
- **Data API URL** — the PostgREST endpoint for your database
- **Auth URL** — your Neon Auth endpoint

### e. Apply RLS

Run `db/data-api-setup.sql` in the SQL Editor:

```sql
GRANT USAGE ON SCHEMA public TO authenticated;
GRANT SELECT, INSERT, UPDATE ON bets TO authenticated;
GRANT USAGE, SELECT ON SEQUENCE bets_id_seq TO authenticated;

ALTER TABLE bets ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS app_rw_bets ON bets;

CREATE POLICY app_rw_bets ON bets
  FOR ALL TO authenticated
  USING      (auth.user_id() = user_id)
  WITH CHECK (auth.user_id() = user_id);
```

### f. Refresh the schema cache

After any schema change, go to **Data API → Refresh schema cache** so PostgREST picks up
the new table structure.

### g. Wire up the app

Paste the **Data API URL** and **Auth URL** into the app's Settings screen. Leave them
as-is if you're using the existing Neon project (the defaults in `src/settings/store.ts`
already point to it).

## 2. How auth and RLS work

```
User enters email + password in Settings
        ↓
App: POST /auth/sign-in/email  (auto-registers on first use)
        ↓
App: GET  /auth/get-session    → set-auth-jwt: <JWT>
        ↓
App: POST /rest/v1/bets        with  Authorization: Bearer <JWT>
        ↓
PostgREST switches to 'authenticated' role
        ↓
Postgres: auth.user_id() = JWT sub claim
        ↓
RLS policy 'app_rw_bets':
  USING (auth.user_id() = user_id)       ← reads only your rows
  WITH CHECK (auth.user_id() = user_id)  ← writes only fill your user_id
```

The `user_id` column has `DEFAULT (auth.user_id())`. The app **never sends `user_id`** in
the request body — Postgres fills it from the JWT automatically. RLS then guarantees you
can only read and write your own rows.

## 3. Authenticate a user against the Data API

This is the exact three-step flow `src/neon/client.ts` implements. You can run it from a
terminal to get a JWT for testing.

Set up environment variables first:

```bash
AUTH_URL="https://<your-auth-endpoint>/auth"
DATA_API_URL="https://<your-data-api-endpoint>/rest/v1"
```

### Step 1 — Sign up (first time) or sign in

**Sign up (first time only):**

```bash
curl -X POST "$AUTH_URL/sign-up/email" -c cookies.txt \
  -H 'Content-Type: application/json' \
  -H 'Origin: http://localhost' \
  -d '{
    "email": "you@example.com",
    "password": "yourpassword",
    "name": "Your Name",
    "callbackURL": "http://localhost"
  }'
```

**Sign in (subsequent times):**

```bash
curl -X POST "$AUTH_URL/sign-in/email" -c cookies.txt \
  -H 'Content-Type: application/json' \
  -H 'Origin: http://localhost' \
  -d '{"email": "you@example.com", "password": "yourpassword"}'
```

The `-c cookies.txt` flag saves the session cookie returned by Neon Auth.

### Step 2 — Exchange cookie for a JWT

```bash
curl "$AUTH_URL/get-session" \
  -b cookies.txt \
  -H 'Origin: http://localhost' \
  -D - -o /dev/null
```

Look for the `set-auth-jwt:` response header. Copy the full token value.

> **JWT TTL:** tokens expire after ~15 minutes. Re-run step 2 (or step 1 + 2) to refresh.
> `client.ts` does this automatically.

### Step 3 — Query the Data API

```bash
JWT="eyJhbGci..."   # paste the token from step 2

# List your bets (RLS returns only your rows)
curl "$DATA_API_URL/bets?select=*&order=placed_at.desc" \
  -H "Authorization: Bearer $JWT"

# Insert a test row (user_id filled automatically by Postgres)
curl -X POST "$DATA_API_URL/bets?on_conflict=user_id,site_key,external_bet_id" \
  -H "Authorization: Bearer $JWT" \
  -H 'Content-Type: application/json' \
  -H 'Prefer: resolution=merge-duplicates' \
  -d '[{"site_key":"test","external_bet_id":"abc-1","status":"open","stake":10}]'
```

### Using the Auth reference UI

Neon Auth exposes an interactive OpenAPI reference UI (powered by Better Auth) at:

```
<AUTH_URL>/reference
```

For example: `https://ep-example.neonauth.us-west-2.aws.neon.tech/neondb/auth/reference`

Use it to explore `sign-up/email`, `sign-in/email`, `get-session`, and all other auth
endpoints interactively in the browser.

## 4. OpenAPI / Swagger schema

### Data API (PostgREST)

The PostgREST root endpoint returns a full OpenAPI JSON schema describing every table,
column, filter, and operator:

```bash
curl "$DATA_API_URL/" \
  -H "Authorization: Bearer $JWT" \
  -H 'Accept: application/openapi+json'
```

This is also accessible via the **Data API** page in the Neon Console — click the table
name to inspect its columns and example queries.

### Auth API

Interactive reference UI at `<AUTH_URL>/reference` (see [above](#using-the-auth-reference-ui)).

## 5. Run with Expo Go (debug)

```bash
npm install
npx expo start
```

Open Expo Go on your iPhone and scan the QR code. No build step, no Xcode required.

**First-time setup:**

1. Open the app → tap **Settings**.
2. Enter your Neon **Data URL** and **Auth URL**.
3. Enter the email + password you want to use as your Neon identity (the app will register
   the account automatically on first sync if it doesn't exist yet).
4. Enter your sportsbook login credentials for each book.
5. Tap **Save**, then tap **Sync** on any book.

## 6. Build for Xcode / standalone

### Option A — Run on device via Xcode

```bash
npx expo prebuild -p ios
open ios/*.xcworkspace
```

Select your device in Xcode and press Run. Config plugins for `expo-notifications` and
`expo-secure-store` are applied automatically by `prebuild`.

Bundle identifier: `com.jacksonkyarger.betloader` (set in `app.json`).

### Option B — EAS Build (cloud)

```bash
npm install -g eas-cli
eas build -p ios
```

This produces an `.ipa` you can distribute via TestFlight or install directly.

## 7. Adding a new book

See [`docs/adding-a-site.md`](docs/adding-a-site.md) for a full walkthrough. Summary:

1. Create `src/books/<sitekey>.ts` with a `BookConfig` (injected JS + `mapBet`).
2. Add it to `src/books/registry.ts`.
3. No database row needed — bets are stored under the user's Neon identity via RLS.
